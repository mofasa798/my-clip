# ffmpeg.md

# FFmpeg Engineering Guide

This document defines how FFmpeg should be used throughout the project.

FFmpeg is the only tool responsible for media processing.

All FFmpeg execution must be centralized through the backend FFmpeg wrapper.

---

# Objectives

The FFmpeg integration should prioritize:

* Correctness
* Performance
* Reliability
* Predictable behavior
* Minimal processing time

---

# Processing Strategy

Always use the following priority:

```text
Stream Copy

↓

GPU Encoding

↓

CPU Encoding
```

This order is mandatory unless explicitly overridden by the user.

---

# Processing Modes

## Mode 1 — Stream Copy

Use whenever possible.

Characteristics:

* No re-encoding
* Fastest export
* Original quality
* Minimal CPU usage

Example use cases:

* Simple clip extraction
* No filters
* No scaling
* No subtitle burn-in

Preferred command pattern:

```text
ffmpeg
-i input.mp4
-ss START
-to END
-c copy
output.mp4
```

---

## Mode 2 — GPU Encoding

Use when re-encoding is required.

Examples:

* Resize
* Filters
* Subtitle burn-in
* Codec conversion

Preferred encoder on target hardware:

```text
h264_amf
```

Future support:

* hevc_amf
* av1_amf (when available)

---

## Mode 3 — CPU Encoding

Fallback only.

Preferred encoder:

```text
libx264
```

CPU encoding should only be used when:

* GPU is unavailable
* GPU encoder fails
* User explicitly requests CPU

---

# Encoder Selection

Preferred order:

```text
Stream Copy

↓

h264_amf

↓

libx264
```

The selection should be automatic.

---

# Command Generation

All commands should be generated by the FFmpeg wrapper.

Business logic must never construct FFmpeg arguments.

Preferred flow:

```text
Clip Service

↓

FFmpeg Wrapper

↓

Argument Builder

↓

exec.CommandContext()

↓

FFmpeg
```

---

# Input Validation

Validate before executing FFmpeg:

* Input file exists
* Output directory exists
* Clip timestamps are valid
* Encoder is supported
* Output filename is valid

Reject invalid requests early.

---

# Timestamp Handling

Always normalize timestamps.

Rules:

* Start >= 0
* End > Start
* End <= Video Duration

Reject invalid ranges.

---

# Output Strategy

Never overwrite user files automatically.

If a filename already exists:

* Generate a unique filename
* Or ask the user for confirmation (future feature)

---

# Progress Reporting

FFmpeg progress should be parsed and emitted as structured events.

Expected event flow:

```text
Export Started

↓

Progress

↓

Completed
```

Include:

* Percentage
* Current timestamp
* Estimated remaining time (when available)

---

# Cancellation

Every FFmpeg process must support cancellation through `context.Context`.

When cancelled:

* Stop FFmpeg immediately
* Clean temporary files
* Emit cancellation event

---

# Temporary Files

Use temporary files only when necessary.

Rules:

* Store in a temporary directory
* Remove automatically
* Never expose to the user

---

# Error Handling

Capture:

* Exit code
* stderr
* Execution time

Return structured errors.

Example:

```text
failed to export clip using h264_amf
```

Avoid generic errors.

---

# Logging

Log only meaningful events:

* Command started
* Encoder selected
* Export completed
* Export cancelled
* Export failed

Do not log every FFmpeg output line.

---

# Filters

Filters should only be added when required.

Common examples:

* scale
* fps
* subtitles
* crop

Do not insert unnecessary filters.

---

# Audio

Preserve original audio whenever possible.

Avoid unnecessary audio re-encoding.

Preferred strategy:

```text
-c:a copy
```

Re-encode only when required by the chosen output format or filters.

---

# Video Quality

When re-encoding:

* Preserve visual quality
* Avoid unnecessary bitrate inflation
* Use encoder defaults unless configured otherwise

Favor sensible defaults over excessive tuning.

---

# Hardware Acceleration

Detect supported hardware during application startup.

Cache detection results.

Do not probe hardware before every export.

---

# Performance

Optimize for:

* Fast startup
* Minimal disk I/O
* Efficient process spawning
* Low memory usage

Avoid multiple FFmpeg processes unless necessary.

---

# Command Execution

Always use:

```go
exec.CommandContext(...)
```

Never use:

```go
exec.Command(...)
```

Context support is mandatory for cancellation and cleanup.

---

# Wrapper Design

The FFmpeg wrapper should expose a small, explicit API.

Example responsibilities:

* Build arguments
* Execute process
* Parse progress
* Return structured result

Avoid exposing raw command generation to higher layers.

---

# Retry Policy

Do not retry failed exports automatically.

If GPU encoding fails:

1. Record the error.
2. Fall back to CPU encoding (if appropriate).
3. Inform the caller of the selected strategy.

Avoid infinite retries.

---

# Testing

Verify:

* Stream Copy
* GPU Encoding
* CPU Fallback
* Invalid timestamps
* Invalid paths
* Cancellation
* Progress parsing
* Error propagation

Each mode should be tested independently.

---

# AI Guidelines

When generating FFmpeg-related code:

* Use the wrapper.
* Keep commands deterministic.
* Validate inputs first.
* Prefer Stream Copy.
* Use GPU only when necessary.
* Use CPU as the final fallback.
* Never execute FFmpeg outside the wrapper.
* Keep command construction centralized.

---

# Future Features

Possible future enhancements:

* AV1 encoding
* HEVC encoding
* Hardware decoding
* Batch export
* Export presets
* Thumbnail generation

These features should remain outside the current implementation until scheduled in the roadmap.

---

# FFmpeg Philosophy

FFmpeg is an implementation detail.

The rest of the application should interact with a clean Go API rather than constructing FFmpeg commands directly.

A reliable wrapper is more valuable than exposing every FFmpeg option.
